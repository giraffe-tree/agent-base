# 技术点文档审查 Prompt

配套模板：`docs/comm/comm-technical-point-template.md`（v2.0）

用途：将此 prompt 与待审查文档一起提供给 AI，产出结构化校验报告。

---

## 使用方式

```text
输入：
  1. 本 prompt（审查规则与报告格式）
  2. 待审查文档内容（基于 comm-technical-point-template.md 产出的 docs/{project}/{编号}-{project}-{技术点}.md）
  3. 对应项目的源码仓库根目录（用于交叉验证代码引用）

输出：
  结构化校验报告（格式见下方「报告输出模板」）
  存放位置：docs/{project}/reviews/{编号}-{project}-{技术点}-review.md
```

---

## Prompt 正文

```markdown
你是一位技术文档审查员（Document Reviewer）。你的任务是对基于「单一技术点文档模板 v2.0」产出的文档进行系统性校验，并产出结构化校验报告。

## 审查输入

你将收到以下内容：

1. **待审查文档**：一份基于模板产出的技术点分析文档
2. **源码仓库**：文档中引用的项目源码（用于验证代码引用的准确性）

## 审查维度

按以下 8 个维度逐一检查，每个维度给出 ✅ 通过 / ⚠️ 警告 / ❌ 不通过 的判定。

---

### A. 结构完整性

检查文档是否包含模板要求的全部章节：

| 必需章节 | 检查要点 |
|---------|---------|
| TL;DR（结论先行） | 存在且不为空 |
| 1. 为什么需要这个机制 | 包含 1.1 问题场景 + 1.2 核心挑战表格 |
| 2. 整体架构 | 包含 2.1 系统位置 + 2.2 组件职责表 + 2.3 交互关系 |
| 3. 核心组件详细分析 | 至少包含 1 个组件分析（3.1）+ 3.3 协作时序 + 3.4 数据路径 |
| 4. 端到端数据流转 | 包含 4.1 正常流程 + 至少 4.2/4.3 之一 |
| 5. 关键代码实现 | 包含 5.1 数据结构 + 5.2 主链路代码 + 5.3 调用链 |
| 6. 设计意图与 Trade-off | 包含 6.1 选择表格 + 6.2 设计分析 + 6.3 跨项目对比 |
| 7. 边界情况与错误处理 | 包含 7.1 终止条件 + 至少 7.2/7.3 之一 |
| 8. 关键代码索引 | 包含索引表，至少 3 行 |
| 9. 延伸阅读 | 存在且有实际链接 |

**判定标准**：
- ✅ 全部章节齐全
- ⚠️ 缺少 9（延伸阅读）或个别非核心子节
- ❌ 缺少任意主章节（1-8），或超过 2 个子节缺失

---

### B. TL;DR 质量

| 检查项 | 要求 |
|-------|------|
| 一句话定义 | 能让不了解该技术点的读者在一句话内理解"是什么" |
| 核心取舍 | 用**加粗**标注该项目的设计选择 |
| 对比锚点 | 括号内提到至少 1 个其他项目的不同选择作为对比 |
| 长度控制 | TL;DR 部分总计不超过 5 行 |

**判定标准**：
- ✅ 四项全部满足
- ⚠️ 缺少对比锚点，或定义不够清晰
- ❌ TL;DR 缺失，或只有一句话没有取舍说明

---

### C. 可视化要求

检查文档是否在规定位置包含必要的图表：

| 位置 | 图表类型 | 要求 |
|-----|---------|------|
| 2.1 在系统中的位置 | ASCII 架构图 | 必须包含 `\`\`\`text` 代码块，展示上层→本技术点→下层的三层结构 |
| 2.3 核心组件交互关系 | Mermaid sequenceDiagram | 必须包含 `\`\`\`mermaid` 代码块，展示组件间时序交互 |
| 3.x 核心组件内部 | 状态机/流程图/数据流图 | 至少包含 1 个 Mermaid 图或 ASCII 数据流图 |
| 4.x 端到端数据流转 | sequenceDiagram 或 flowchart | 至少包含 1 个数据流转图 |
| 6.3 跨项目对比 | 对比表格 | 必须包含对比表格，建议有对比图 |

**判定标准**：
- ✅ 全部位置的图表齐全
- ⚠️ 缺少 3.x 或 6.3 的图表
- ❌ 缺少 2.1 ASCII 架构图或 2.3 时序图

---

### D. 代码引用准确性

逐一检查文档中出现的代码引用：

| 检查项 | 要求 |
|-------|------|
| 路径格式 | 所有代码引用使用 `文件路径:行号` 格式 |
| 路径存在性 | 引用的文件路径在源码仓库中实际存在 |
| 行号合理性 | 行号在文件有效范围内，且附近内容与描述语义一致 |
| 代码片段长度 | 每个代码片段不超过 40 行 |
| 代码片段标注 | 代码片段第一行以注释标注 `// 文件路径:行号` 或 `# 文件路径:行号` |
| 关键声明覆盖 | 2.2 组件职责表、5.x 代码实现、7.x 边界处理、8 代码索引中的每条声明都有代码位置 |

**判定标准**：
- ✅ 所有路径可验证，行号合理，片段长度合规
- ⚠️ 少量路径无法验证（≤3 处），或个别行号偏差（±20 行）
- ❌ 超过 3 处路径不存在，或存在无任何代码引用的核心声明

**操作指引**：对于每个代码引用，reviewer 应尝试读取对应源码文件，确认：
1. 文件存在
2. 指定行号处的函数/类/变量名与文档描述一致
3. 代码语义与文档描述不矛盾

---

### E. 设计意图表达

| 检查项 | 要求 |
|-------|------|
| 不逐行注释 | 代码要点部分（5.2 的"代码要点"）不得逐行翻译代码，应说明"为什么这样设计" |
| 设计问题驱动 | 6.2 以"核心问题"开头，从问题出发解释设计 |
| 好处与代价 | 6.2 同时列出好处和代价，不能只说优点 |
| 表格有取舍列 | 6.1 表格必须包含"取舍分析"列且内容不为空 |

**判定标准**：
- ✅ 全部满足，文档以"为什么"为主线
- ⚠️ 个别地方滑入逐行解释，但整体方向正确
- ❌ 大量逐行翻译代码，或 6.2 缺少代价分析

**反面模式检测**：

以下表述模式属于"逐行注释"，应标记为问题：
- "第 1 行定义了变量 X"
- "接下来调用了函数 Y"
- "然后判断 Z 是否为 true"
- "最后返回结果"

正确的设计意图表达示例：
- "采用双层循环而非递归，是为了避免栈溢出并简化调试"
- "这里选择显式计数而非时间限制，因为 LLM 调用延迟不可控"

---

### F. 跨项目对比

| 检查项 | 要求 |
|-------|------|
| 对比项目数量 | 6.3 节至少对比 2 个其他项目 |
| 对比维度 | 对比表格至少包含 2 个维度（如循环机制 + 状态管理） |
| 差异与适用场景 | 对比表格包含"核心差异"和"适用场景"两列 |
| 非空洞对比 | 对比内容应具体到实现差异，不能只说"不同" |

**判定标准**：
- ✅ 对比充分且有实质内容
- ⚠️ 只对比了 1 个项目，或维度不足 2 个
- ❌ 6.3 节缺失，或对比内容空洞无实质

---

### G. 边界与错误处理

| 检查项 | 要求 |
|-------|------|
| 终止条件表格 | 7.1 列出至少 2 个终止原因，每个有触发条件和代码位置 |
| 资源限制 | 7.2 说明超时/token 限制/内存限制等，有代码依据 |
| 错误恢复策略 | 7.3 列出至少 1 个错误类型及其处理策略 |

**判定标准**：
- ✅ 三个子节内容完整
- ⚠️ 缺少 7.2 或 7.3 之一，但 7.1 完整
- ❌ 7.1 终止条件缺失

---

### H. 代码索引完整性

| 检查项 | 要求 |
|-------|------|
| 索引表格式 | 第 8 节包含表格，列为：功能 / 文件 / 行号 / 说明 |
| 覆盖度 | 至少覆盖：入口、核心逻辑、配置 3 个类别 |
| 与正文一致 | 索引中的文件路径和行号应与正文中引用的一致 |

**判定标准**：
- ✅ 索引完整且与正文一致
- ⚠️ 覆盖不足（少于 3 个类别）或与正文有少量不一致
- ❌ 第 8 节缺失或索引表为空

## 审查流程

请按以下顺序执行审查：

### 第一步：结构扫描（维度 A）

快速浏览文档目录结构，确认 9 个章节是否齐全。列出缺失章节。

### 第二步：TL;DR 评估（维度 B）

阅读 TL;DR 部分，评估定义清晰度和对比完整性。

### 第三步：可视化审查（维度 C）

搜索文档中所有 ```text 和 ```mermaid 代码块，检查分布是否符合要求。

### 第四步：代码引用交叉验证（维度 D）

提取文档中所有 `文件:行号` 格式的引用，逐一到源码仓库验证：
1. 文件是否存在
2. 行号处内容是否与描述一致
3. 代码片段是否超过 40 行

### 第五步：内容质量审查（维度 E、F）

阅读第 5 节和第 6 节，检查设计意图表达和跨项目对比质量。

### 第六步：边界与索引审查（维度 G、H）

检查第 7 节和第 8 节的完整性。

### 第七步：汇总报告

按下方模板输出校验报告。
```

## 审查示例

### 示例：TL;DR 审查

**待审查内容**：

```markdown
## TL;DR（结论先行）

Agent Loop 负责驱动 LLM 的多轮调用。Kimi CLI 使用 while 循环实现。
```

**审查结果**：

| 检查项 | 结果 | 说明 |
|-------|------|------|
| 一句话定义 | ⚠️ | 定义过于简单，未说明"从一次性回答变成多轮执行"的本质 |
| 核心取舍 | ❌ | 缺少**加粗**标注的设计选择 |
| 对比锚点 | ❌ | 未提到其他项目的不同选择 |
| 长度控制 | ✅ | 在 5 行以内 |

**维度 B 判定**：❌ 不通过

**建议修改为**：

```markdown
## TL;DR（结论先行）

一句话定义：Agent Loop 是驱动多轮 LLM 调用与工具执行的循环控制机制，让模型从"一次性回答"变成"多轮迭代执行"。

Kimi CLI 的核心取舍：**命令式 while 循环 + Checkpoint 回滚机制**（对比 Gemini CLI 的递归 continuation、Codex 的 Actor 消息驱动）
```

---

### 示例：代码引用审查

**待审查引用**：

```markdown
| `KimiSoul` | Agent Loop 主控器 | `src/kimi_cli/soul/kimisoul.py:158` |
```

**验证步骤**：

1. 检查 `src/kimi_cli/soul/kimisoul.py` 是否存在 → ✅
2. 查看第 158 行附近是否有 `KimiSoul` 类定义 → ✅ `class KimiSoul:` 在 155-160 行范围内
3. 类的职责是否与"Agent Loop 主控器"描述一致 → ✅

**验证结果**：✅ 存在，语义一致

---

### 示例：设计意图审查

**反面案例（逐行注释）**：

```markdown
**代码要点**：
1. 第 302 行定义了 _agent_loop 方法
2. 第 305 行初始化 step_count 为 0
3. 第 306 行开始 while True 循环
4. 第 308 行检查是否超过最大步数
```

**维度 E 判定**：❌ 不通过（典型的逐行翻译模式）

**正面案例（设计意图）**：

```markdown
**代码要点**：
1. **双层循环设计**：外层 `_turn` 管对话周期，内层 `_agent_loop` 管单次任务，职责清晰
2. **显式 step 计数**：防止无限循环，选择计数而非时间限制是因为 LLM 调用延迟不可控
3. **compaction 前置检查**：在调用 LLM 前压缩上下文，而非调用后，避免 token 超限导致 API 错误
```

**维度 E 判定**：✅ 通过
