# 一年后的 Coding Agent 应该长什么样？

## 一句话结论

一年后（2027 年初）有竞争力的 Coding Agent 需要 **14 项核心能力**，其中 **9 项可以自己构建**，**5 项依赖外部环境成熟**。最关键的自建能力是**语义代码理解层**和**结构化验证循环**，最关键的外部依赖是**模型推理能力跃升**和**MCP 生态成熟**。

## 1. 目标画像

### 1.1 一年后的 Agent 应该能做什么（vs 做不到什么）

当前（2026 年初）主流 Agent 的能力边界大致是：

```
能做：单文件修改、简单 bug 修复、模板化功能添加
勉强：跨 2-3 个文件的修改、简单重构
做不到：跨模块功能开发、架构级重构、需要深度业务理解的任务
```

一年后（2027 年初），目标能力边界应该是：

```
能做：跨 5-10 个文件的功能开发、中等复杂度重构、测试驱动的 bug 修复
勉强：跨模块功能开发（需要人工辅助分解任务）
做不到：全新系统架构设计、跨仓库协调（但已有初步能力）
```

### 1.2 量化目标

| 指标 | 当前 (2026) | 目标 (2027) | 提升倍数 |
|------|------------|------------|---------|
| 单任务涉及文件数 | 1-3 | 5-15 | 3-5x |
| 首次正确率（单文件） | ~65% | ~85% | 1.3x |
| 首次正确率（跨文件） | ~20% | ~55% | 2.7x |
| 单任务平均耗时 | 5-15 min | 3-8 min | 1.5-2x |
| 需要人工干预的比例 | ~60% | ~30% | 2x |
| 可处理的代码库规模 | <5万行 | <30万行 | 6x |

## 3. 自建能力：9 项你必须自己造的轮子

### ① Agent Loop 2.0 —— 强化版核心循环

**为什么需要**：当前 5 个主流项目的 Agent Loop 都是简单的 `while(true)` + 工具调用。一年后需要支持更复杂的控制流。

**当前最佳实践参考**：

| 现有项目 | 可借鉴的点 | 不足之处 |
|---------|-----------|---------|
| Codex | Actor 模型，Event Channel，CancellationToken | 无循环检测，无自适应 |
| Gemini CLI | 递归 continuation，Loop Detection | 递归深度受栈限制 |
| Kimi CLI | Checkpoint + D-Mail 回滚 | 回滚仅限上下文，不含文件系统 |
| OpenCode | while(true) + 任务分支 (subtask/compaction/normal) | 分支逻辑硬编码 |
| SWE-agent | Thought-Action 解耦 | 太简单，无高级控制 |

**你应该构建的 Agent Loop 2.0**：

```
┌───────────────────────────────────────────────────────────┐
│  Agent Loop 2.0                                            │
│                                                            │
│  ┌──────────────────────────────────────────────────────┐ │
│  │  决策层 (新增)                                        │ │
│  │  ├── 计划评估器：当前执行是否符合计划？                │ │
│  │  ├── 进度追踪器：完成了多少，还剩多少？                │ │
│  │  ├── 策略选择器：继续/修改计划/回滚/求助？             │ │
│  │  └── 资源监控器：token/成本/时间 预算还剩多少？        │ │
│  └──────────────────────────────────────────────────────┘ │
│                        │                                   │
│                        ▼                                   │
│  ┌──────────────────────────────────────────────────────┐ │
│  │  执行层 (增强)                                        │ │
│  │  ├── LLM 推理（同现有）                               │ │
│  │  ├── 工具执行（同现有，+ 并发）                       │ │
│  │  ├── 验证器执行（新增：类型检查/lint/测试）           │ │
│  │  └── 结果聚合（新增：合并多个工具/验证结果）          │ │
│  └──────────────────────────────────────────────────────┘ │
│                        │                                   │
│                        ▼                                   │
│  ┌──────────────────────────────────────────────────────┐ │
│  │  反馈层 (新增)                                        │ │
│  │  ├── 结构化反馈：不只是文本，而是 {错误类型, 位置,    │ │
│  │  │               相关代码, 修复建议}                   │ │
│  │  ├── 上下文增强：自动检索相关的类型定义/接口/测试     │ │
│  │  └── 历史去重：避免重复注入已知信息                   │ │
│  └──────────────────────────────────────────────────────┘ │
│                                                            │
│  终止条件 (增强)：                                         │
│  ├── 自然完成：模型无 tool call + 验证器全部通过          │
│  ├── 预算耗尽：token/成本/时间 达到限制                   │
│  ├── 循环检测：连续 N 步相似操作（参考 Gemini CLI）       │
│  ├── 回滚触发：关键验证持续失败，回滚到 checkpoint        │
│  └── 求助升级：Agent 判断自己无法解决，精确描述卡点       │
└───────────────────────────────────────────────────────────┘
```

**关键实现建议**：
- 从 Kimi CLI 借鉴 Checkpoint 机制，但扩展到文件系统级别（git stash/branch）
- 从 Gemini CLI 借鉴 Loop Detection，但升级为语义级别（不只是模式匹配）
- 从 OpenCode 借鉴任务分支，但增加动态分支创建能力

**工作量评估**：3-4 周核心开发 + 2 周调优

### ② 语义代码理解层 —— 最关键的差异化能力

**为什么这是最重要的自建能力**：

当前所有 5 个 Agent 都把代码当纯文本处理。这意味着：
- `grep "getUserById"` 找不到 `const getUser = getUserById` 这种别名
- 修改函数签名后不知道哪些调用方需要同步修改
- 不了解类型约束，生成的代码经常有类型错误

语义理解层让 Agent 从"文本编辑器"进化为"IDE"。

**架构设计**：

```
┌──────────────────────────────────────────────────────────────┐
│  Semantic Code Understanding Layer                            │
│                                                                │
│  离线构建 (项目初始化时，增量更新)                               │
│  ┌──────────────────────────────────────────────────────────┐ │
│  │                                                            │ │
│  │  代码库                                                    │ │
│  │    │                                                       │ │
│  │    ├──▶ tree-sitter 解析 ──▶ AST 索引                     │ │
│  │    │      (所有主流语言)       文件→函数→类→接口 的层级     │ │
│  │    │                                                       │ │
│  │    ├──▶ LSP 分析 ──▶ 类型索引                             │ │
│  │    │      (tsserver/rust-analyzer/pyright)                  │ │
│  │    │      每个符号的类型、定义位置、引用列表                │ │
│  │    │                                                       │ │
│  │    ├──▶ 依赖分析 ──▶ 依赖图                               │ │
│  │    │      import/require/use 关系                          │ │
│  │    │      模块 → 模块 的依赖和被依赖                       │ │
│  │    │                                                       │ │
│  │    └──▶ 嵌入模型 ──▶ 语义向量索引                         │ │
│  │           (可选，依赖外部能力 ⑬)                           │ │
│  │           每个函数/类的语义表示                             │ │
│  └──────────────────────────────────────────────────────────┘ │
│                                                                │
│  在线查询 (Agent 推理时按需调用)                                │
│  ┌──────────────────────────────────────────────────────────┐ │
│  │                                                            │ │
│  │  工具 API (注册为 Agent 可调用的 Tool)：                    │ │
│  │                                                            │ │
│  │  find_references(symbol)                                   │ │
│  │    → 谁在用这个函数/类/变量？                              │ │
│  │                                                            │ │
│  │  get_type_info(symbol)                                     │ │
│  │    → 这个符号的完整类型信息？                              │ │
│  │                                                            │ │
│  │  get_call_chain(function)                                  │ │
│  │    → A 调用 B，B 调用 C... 的调用链是什么？                │ │
│  │                                                            │ │
│  │  impact_analysis(file, change_description)                 │ │
│  │    → 如果修改这个文件，哪些其他文件可能受影响？            │ │
│  │                                                            │ │
│  │  find_similar_code(description)                            │ │
│  │    → 项目中有没有类似功能的代码可以参考？                  │ │
│  │                                                            │ │
│  │  get_module_summary(path)                                  │ │
│  │    → 这个目录/模块是做什么的？包含哪些关键接口？           │ │
│  │                                                            │ │
│  └──────────────────────────────────────────────────────────┘ │
└──────────────────────────────────────────────────────────────┘
```

**技术选型建议**：

| 组件 | 推荐方案 | 替代方案 | 理由 |
|------|---------|---------|------|
| AST 解析 | tree-sitter | Babel/swc (JS-only) | 多语言统一接口，增量解析 |
| 类型分析 | 各语言原生 LSP | 自建类型推断 | LSP 已是工业标准 |
| 依赖图 | 自建（基于 AST） | madge (JS-only) | 需要跨语言统一 |
| 语义搜索 | 代码嵌入模型 + 向量 DB | 关键词搜索 | 依赖外部能力 ⑬ |
| 存储 | SQLite + 内存缓存 | 图数据库 | 简单可靠，参考 Codex/OpenCode |

**工作量评估**：4-6 周（tree-sitter + LSP 基础），持续迭代

### ③ 结构化验证循环 —— 从"猜测"到"证明"

**核心思想**：每次代码修改后，不是让 Agent 自己判断"看起来对不对"，而是用确定性工具验证。

```
当前：
  Agent 修改代码 → Agent 读修改后的文件 → Agent 自行判断是否正确
                                         (经常判断错误)

一年后：
  Agent 修改代码 ──┬──▶ TypeChecker ──▶ {ok} 或 {errors: [...]}
                   ├──▶ Linter ──▶ {warnings: [...]}
                   ├──▶ TestRunner ──▶ {passed: N, failed: [...]}
                   └──▶ BuildSystem ──▶ {success} 或 {errors: [...]}
                                │
                                ▼
                         结构化聚合结果
                         {
                           type_errors: [{file, line, msg, related_type}],
                           lint_warnings: [{rule, file, line, fix_suggestion}],
                           test_failures: [{name, assertion, stack, related_code}],
                           build_errors: [{step, msg}]
                         }
                                │
                                ▼
                   Agent 基于精确信息修复（而非猜测）
```

**需要实现的验证器**：

| 验证器 | 触发时机 | 输出格式 | 实现难度 |
|--------|---------|---------|---------|
| 类型检查 | 每次文件修改后 | 错误位置 + 类型信息 + 相关定义 | 中（调用 tsc/pyright） |
| Lint 检查 | 每次文件修改后 | 规则 + 位置 + 自动修复建议 | 低（调用 eslint/ruff） |
| 单元测试 | 每次功能修改后 | 通过/失败 + 断言详情 + 覆盖率 | 中（需要解析测试框架输出） |
| 构建检查 | 关键修改后 | 构建成功/失败 + 错误详情 | 低（调用 build 命令） |
| 依赖检查 | 修改 import/依赖后 | 循环依赖/缺失依赖 | 低（基于依赖图） |

**关键设计决策**：

```
问题：验证器应该在每次工具调用后自动运行，还是由 Agent 主动调用？

答案：混合模式
  - 类型检查和 lint → 每次文件修改后自动运行（成本低，价值高）
  - 测试运行 → Agent 主动调用（成本高，需要选择时机）
  - 构建检查 → 完成一组相关修改后自动运行（中等频率）

参考：这类似 IDE 中的实时检查 vs 手动运行测试的关系。
```

**工作量评估**：2-3 周基础框架 + 每个验证器 0.5-1 周

### ④ 自适应规划引擎

**当前 5 个项目规划能力的局限**（详见 `docs/comm/comm-plan-and-execute.md`）：

所有项目的"规划"都是一次性的文本生成。没有项目实现了真正的**动态规划**。

**你应该构建的规划引擎**：

```
┌──────────────────────────────────────────────────────────────┐
│  Plan Engine                                                   │
│                                                                │
│  数据结构：Plan = Tree<PlanNode>                               │
│                                                                │
│  PlanNode {                                                    │
│    id: string                                                  │
│    description: string          // 做什么                      │
│    status: pending | running | done | failed | skipped         │
│    children: PlanNode[]         // 子步骤                      │
│    alternatives: PlanNode[][]   // 替代方案列表                │
│    dependencies: string[]       // 依赖哪些其他节点完成        │
│    validation: Validator[]      // 完成后用什么验证            │
│    estimated_cost: TokenBudget  // 预估消耗                    │
│  }                                                             │
│                                                                │
│  操作：                                                        │
│  ├── create_plan(task) → Plan       // 初始规划                │
│  ├── refine_node(id, new_info)      // 基于新信息细化节点      │
│  ├── add_alternative(id, alt_plan)  // 添加替代方案            │
│  ├── switch_to_alternative(id, n)   // 切换到替代方案 n        │
│  ├── mark_done(id, result)          // 标记完成 + 记录结果     │
│  ├── mark_failed(id, reason)        // 标记失败 + 触发回溯     │
│  └── get_next_actionable()          // 返回下一个可执行节点    │
│                                                                │
│  Agent Loop 集成：                                             │
│  while (plan.has_pending_nodes()):                             │
│      node = plan.get_next_actionable()                         │
│      result = execute(node)                                    │
│      if result.success:                                        │
│          plan.mark_done(node.id, result)                       │
│      else:                                                     │
│          if node.alternatives.length > 0:                      │
│              plan.switch_to_alternative(node.id, 0)            │
│          else:                                                 │
│              plan.mark_failed(node.id, result.error)           │
│              plan = llm.revise_plan(plan, result.error)        │
│                                                                │
└──────────────────────────────────────────────────────────────┘
```

**工作量评估**：3-4 周

### ⑤ 智能上下文管理

**借鉴现有项目的最佳实践，然后超越**：

| 现有方案 | 来源 | 保留 | 改进 |
|---------|------|------|------|
| LLM 摘要压缩 | Codex | ✅ 生成语义摘要 | 改为结构化摘要（保留代码路径和类型信息） |
| Checkpoint 回滚 | Kimi CLI | ✅ 上下文快照 + 恢复 | 扩展到文件系统（git stash） |
| Compaction + Prune | OpenCode | ✅ 双层压缩策略 | 增加优先级：验证结果 > 工具输出 > 探索历史 |
| 三层分层内存 | Gemini CLI | ✅ 项目/用户/会话分层 | 增加"代码理解缓存"层 |

**你应该构建的上下文管理器**：

```
上下文 = 固定区 + 动态区 + 缓存区

固定区 (始终保留，不压缩)：
  ├── 系统提示词
  ├── 项目级 Memory (RULES.md / CLAUDE.md 等)
  ├── 当前计划 (Plan 结构)
  └── 最近的验证结果

动态区 (智能管理)：
  ├── 对话历史 (按相关性压缩，而非时间顺序截断)
  ├── 工具调用结果 (大输出自动摘要)
  └── 探索过程记录 (成功路径保留，失败路径摘要)

缓存区 (按需加载)：
  ├── 代码理解层的查询结果 (AST/类型/依赖信息)
  ├── 相关文件内容 (通过语义检索)
  └── 历史会话的经验摘要
```

**工作量评估**：3-4 周

### ⑥ 多 Agent 协作

**一年后可行的协作范围**：不要试图构建完全自主的多 Agent 系统。目标是"1 个主 Agent + 2-3 个专业子 Agent"。

```
最小可行的多 Agent 架构：

  ┌─────────────────────────────────────────────────┐
  │  Main Agent (主 Agent)                           │
  │  职责：理解需求 → 分解任务 → 协调子 Agent → 集成 │
  │  模型：最强模型 (高推理能力)                      │
  └───────────┬─────────────────────────────────────┘
              │
```

### ⑦ 项目级持久记忆

**目标**：Agent 使用一个项目越久，对它越"熟悉"。

```
记忆层级：

Level 1: 项目结构记忆 (自动构建，第一次交互时)
  "这个项目用 TypeScript，monorepo 结构，用 pnpm，
   主要模块在 packages/ 下，测试用 vitest"

Level 2: 代码模式记忆 (逐步积累)
  "这个项目的 API 都定义在 src/api/，用 Zod 做验证，
   错误处理统一用 AppError 类"

Level 3: 经验记忆 (从成功/失败中学习)
  "上次修改 auth 模块时，忘了更新 middleware 中的类型，
   导致编译失败。auth 相关修改要检查 middleware/"

Level 4: 偏好记忆 (从用户反馈中学习)
  "这个用户偏好 functional style，不喜欢 class，
   代码注释用英文，commit message 用中文"

存储方式：
  .agent/
  ├── project-memory.json    # Level 1-2: 结构化项目知识
  ├── experience-log.jsonl   # Level 3: 经验日志（追加写）
  ├── preferences.json       # Level 4: 用户偏好
  └── code-index/            # 代码理解层的持久化索引
```

**参考 Gemini CLI 的三层内存**：Gemini CLI 的 `GEMINI.md` 发现机制（项目根 + 子目录 + 全局）可以作为持久记忆的载体格式。

### ⑧ 安全沙箱体系

## 4. 外部依赖：5 项需要等环境成熟的能力

### ⑩ 基础模型推理能力（成熟概率：85%）

**当前状态**：GPT-4 / Claude-3.5 / Gemini 2.0 级别，中等复杂度推理

**一年后预期**：
- 推理深度提升 2-3x（更深的 CoT、更好的计划能力）
- 代码生成首次正确率从 ~65% 提升到 ~80%+
- 更好的指令遵从（减少"自由发挥"偏离需求的情况）

**你的对冲策略**：
- 不要把所有赌注押在"模型会更强"上
- 用**结构化验证循环（能力③）**弥补模型不够强的问题
- 用**自适应规划（能力④）**让较弱的模型也能完成复杂任务

**架构兼容设计**：
```
设计为模型无关架构，支持热切换：
  - 主推理用最强模型 (GPT-5 / Claude-4 / Gemini 3.0 等)
  - 验证/摘要用快速模型 (GPT-4o-mini / Claude-3-haiku 等)
  - 支持 fallback：主模型失败时自动切换备选模型

参考 Gemini CLI 的模型路由机制和 Codex 的多模型支持。
```

### ⑪ 长上下文窗口（成熟概率：80%）

**当前状态**：128K-1M token（实际有效使用约 60-70%）

**一年后预期**：2M-10M token 可用窗口

**你的对冲策略**：
- **不要依赖长上下文来解决所有问题**
- 即使有 10M token 窗口，也不可能塞入一个完整的大型项目
- **语义代码理解层（能力②）+ RAG** 是更可靠的路径
- 长上下文是"锦上添花"，语义理解是"雪中送炭"

```
设计原则：
  - 用 RAG 精确检索相关代码片段（100-500 行），而非暴力塞入整个文件
  - 用上下文窗口容纳：系统提示 + 计划 + 对话历史 + RAG 检索结果
  - 长上下文的价值在于保留更长的对话历史，而非替代 RAG
```

### ⑫ MCP 生态成熟（成熟概率：60%）

**当前状态**：MCP 协议已发布，Codex/Gemini CLI/OpenCode 已支持，但第三方 MCP Server 生态仍在早期

**一年后预期**：
- 主流开发工具（数据库、API 测试、部署）会有官方 MCP Server
- MCP Server 的发现和安装会更简便
- 但长尾需求仍需自建

**你的对冲策略**：
```
两条腿走路：
1. 核心工具自建（文件操作、Shell、代码搜索、验证器）
   → 不依赖 MCP，自己完全控制
   
2. 可扩展能力通过 MCP（数据库查询、API 测试、云服务操作）
   → 设计好 MCP 集成接口，生态成熟后即插即用
   
架构：
  ┌─────────────────────────────────────────┐
  │  Tool Registry                           │
  │  ├── Built-in Tools (自建，不可替代)    │
  │  │   ├── file_read / file_write         │
  │  │   ├── semantic_search (能力②)        │
  │  │   ├── type_check / lint (能力③)      │
  │  │   └── bash / shell                    │
  │  │                                       │
  │  └── MCP Tools (可选，即插即用)         │
  │      ├── database_query                  │
  │      ├── api_test                        │
  │      └── deployment                      │
  └─────────────────────────────────────────┘

参考 Codex 的 McpClientManager 和 Gemini CLI 的三层工具来源架构。
```

### ⑬ 代码嵌入模型（成熟概率：70%）

**当前状态**：通用嵌入模型对代码的理解有限，专用代码嵌入模型（CodeBERT、StarCoder Embedding）质量在提升但未达理想

**一年后预期**：会有更好的代码嵌入模型，但可能仍不完美

**你的对冲策略**：
```
语义搜索不是只有嵌入一条路：

Plan A: 嵌入模型 (如果外部能力成熟)
  → 高质量代码嵌入 + 向量搜索 → 最优

Plan B: 混合搜索 (如果嵌入不够好)
  → tree-sitter AST 特征 + 关键词搜索 + LLM 重排序 → 够用

Plan C: LLM-powered 搜索 (兜底)
  → 让 LLM 生成搜索查询 + grep/ast 搜索 → 基本可用

建议：先实现 Plan B，预留 Plan A 的接口。
```

### ⑭ 低成本推理（成熟概率：75%）

**当前状态**：每个复杂 task 成本约 $0.5-5

**一年后预期**：成本降低 5-10x

**你的对冲策略**：
```
分级模型使用策略：

高成本模型 (主推理，只在关键决策时用)：
  - 任务理解和分解
  - 关键代码修改决策
  - 失败后的诊断和策略调整

低成本模型 (辅助任务，大量使用)：
  - 上下文压缩/摘要
  - 验证结果解读
  - 代码格式化和文档生成
  - 子 Agent 的简单任务

零成本操作 (确定性工具，无限使用)：
  - 类型检查、lint（能力③）
  - AST 解析、依赖分析（能力②）
  - 测试运行、构建检查

目标：将 LLM 调用次数减少 40%，用确定性工具替代。
```

| 错误处理 | Kimi CLI | 四层错误继承体系 | `kimi-cli/packages/kosong/src/kosong/tooling/error.py` |
| 流式输出 | Gemini CLI | Event generator yield* | `packages/core/src/core/turn.ts` |
| 安全沙箱 | Codex | Landlock/Seatbelt | `codex-rs/core/src/sandbox/` |
| 多 Agent | OpenCode | Build/Plan/Explore Agent | `packages/opencode/src/agent/agent.ts` |
| 自动迭代 | Kimi CLI | Ralph Loop | `kimi-cli/src/kimi_cli/agent/soul.py` |

### 不应该"抄"的（需要自己创新的部分）

| 领域 | 为什么不能直接复用 | 需要的创新 |
|------|-----------------|-----------|
| 语义代码理解 | 5 个项目都没有 | 全新构建 |
| 结构化验证循环 | 都用 bash 跑测试 | 需要深度集成 |
| 自适应规划 | 都是静态规划 | 需要设计动态计划数据结构 |
| 项目级记忆 | 都只有会话级记忆 | 需要设计经验积累机制 |
| Agent 间共享理解 | 多 Agent 各自独立 | 需要共享代码理解层 |

## 8. 风险与对冲

### 风险矩阵

| 风险 | 概率 | 影响 | 对冲策略 |
|------|------|------|---------|
| 模型推理提升不及预期 | 20% | 高 | 结构化验证补偿模型不足；自适应规划降低对单次推理的依赖 |
| 长上下文窗口未到 10M | 30% | 中 | RAG + 语义理解是主路径，长上下文是加分项 |
| MCP 生态发展缓慢 | 40% | 低 | 核心工具自建，MCP 只做可选扩展 |
| 代码嵌入模型质量不够 | 30% | 中 | 混合搜索（AST + 关键词 + LLM 重排序）作为 Plan B |
| 成本降低不到 10x | 25% | 中 | 分级模型使用 + 确定性工具替代 LLM 调用 |
| tree-sitter 某些语言支持差 | 15% | 低 | 优先支持 TS/Python/Rust/Go/Java，长尾语言降级为文本模式 |
| 竞争对手先发布类似产品 | 50% | 高 | 语义理解层 + 持久记忆是护城河；速度是关键 |

### 最坏情况下的最小可行产品

如果外部依赖全部不及预期，你的 Agent 至少应该是：

```
最小可行产品 = Agent Loop 2.0 + 结构化验证循环 + 基础安全

即使模型不变强，即使上下文窗口不变大，即使 MCP 生态不成熟，
"修改代码后自动验证（类型+lint+测试）并基于精确反馈修复"
这一能力就已经比当前所有 Agent 强一个台阶。

这是你的底线，也是第一个里程碑。
```
