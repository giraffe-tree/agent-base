# Claude Code 分层记忆与上下文管理

**结论**：Claude Code 的记忆系统由**两大部分**构成——**6 层持久化记忆**（会话启动时加载）和**3 层上下文压缩**（会话运行中管理）。前者解决「记住什么」，后者解决「装不下了怎么办」。二者协同工作，让 Agent 在 200K token 的有限窗口内保持长期记忆和任务连续性。

---

## 1. 全景架构

```
┌──────────────────────────────────────────────────────────────┐
│                    Claude Code 记忆系统                        │
│                                                              │

│  │                                                       │    │
│  │  L1  Managed Policy     企业/组织级策略               │    │
│  │  L2  User Memory        用户全局偏好                  │    │
│  │  L3  Project Memory     项目团队共享指令              │    │
│  │  L4  Project Rules      模块化规则（按路径匹配）      │    │
│  │  L5  Local Memory       个人项目偏好（不入 Git）      │    │
│  │  L6  Auto Memory        Claude 自动发现的模式         │    │

│  └───────────────────────────────────────────────────────┘    │
│                          │                                    │
│                          ▼ 加载到                              │

│  │                                                       │    │

│  │  + 对话历史                                           │    │
│  │  + 工具调用 & 结果                                    │    │
│  │                                                       │    │
│  │  ┌─── 3 层压缩系统 ──────────────────────────────┐    │    │
│  │  │  Micro-compaction   大输出转存磁盘             │    │    │
│  │  │  Auto-compaction    ~83.5% 时自动压缩          │    │    │
│  │  │  Manual-compaction  用户 /compact 手动触发      │    │    │
│  │  └────────────────────────────────────────────────┘    │    │
│  │                                                       │    │
│  └───────────────────────────────────────────────────────┘    │
│                                                              │
└──────────────────────────────────────────────────────────────┘
```

## 2. 六层持久化记忆详解

### 2.1 完整层次表

| 层次 | 名称 | 文件位置 | 作用域 | 谁维护 | 入 Git |
|------|------|---------|--------|--------|--------|
| L1 | Managed Policy | `/etc/claude-code/CLAUDE.md` (Linux) | 组织全员 | IT/DevOps | N/A |
| L2 | User Memory | `~/.claude/CLAUDE.md` | 用户所有项目 | 用户 | 否 |
| L3 | Project Memory | `./CLAUDE.md` 或 `./.claude/CLAUDE.md` | 项目团队 | 团队 | 是 |
| L4 | Project Rules | `./.claude/rules/*.md` | 项目（按路径） | 团队 | 是 |
| L5 | Local Memory | `./CLAUDE.local.md` | 个人+项目 | 用户 | 否（自动 gitignore） |
| L6 | Auto Memory | `~/.claude/projects/<hash>/memory/` | 个人+项目 | Claude 自动 | 否 |

### 2.2 逐层剖析

#### L1：Managed Policy（组织策略层）

```
位置:
  macOS:   /Library/Application Support/ClaudeCode/CLAUDE.md
  Linux:   /etc/claude-code/CLAUDE.md
  Windows: C:\Program Files\ClaudeCode\CLAUDE.md
```

这是最顶层的策略，由企业 IT/DevOps 统一部署。典型内容：

```markdown
# 公司 AI 编码策略
- 禁止在代码中硬编码 API Key
- 所有数据库操作必须使用参数化查询
- 不得修改 production 分支
- 代码提交必须包含 JIRA ticket 号
```

**设计意图**：让企业能在不修改任何项目代码的情况下，对所有开发者的 Claude Code 行为施加安全约束。类似于浏览器的企业组策略。

#### L2：User Memory（用户偏好层）

```
位置: ~/.claude/CLAUDE.md
```

跨所有项目的个人偏好。典型内容：

```markdown
# 我的偏好
- response in 中文
- 使用 vim 键位习惯描述操作
- 偏好函数式风格，避免 class
- commit message 用英文，格式：type(scope): description
```

**设计意图**：一次配置，所有项目生效。不需要每个项目都重复声明个人习惯。

#### L3：Project Memory（项目共享层）

```
位置: ./CLAUDE.md 或 ./.claude/CLAUDE.md
```

团队共享的项目级指令，提交到 Git。这是最常用的记忆层。典型内容：

```markdown
# 项目说明
本项目使用 TypeScript + React + Prisma
测试框架: vitest
包管理: pnpm

## 代码规范
- 组件使用 PascalCase
- hooks 以 use 开头
- API 路由在 src/api/ 下

## 常用命令
- pnpm dev: 启动开发服务器
- pnpm test: 运行测试
- pnpm lint: 代码检查
```

**设计意图**：项目知识的「团队共识」，每个团队成员的 Claude Code 都能读取。

#### L4：Project Rules（模块化规则层）

```
位置: ./.claude/rules/*.md
```

比 L3 更细粒度的规则文件，支持 **glob 路径匹配**。典型目录结构：

```
.claude/rules/
├── testing.md          # 测试相关规则
├── api-design.md       # API 设计规范
├── database.md         # 数据库操作规范
└── frontend.md         # 前端代码规范（可配置仅匹配 src/components/**）
```

每个规则文件可以在头部指定匹配范围：

```markdown
---
globs: src/components/**/*.tsx
---
# 前端组件规范
- 所有组件必须导出 Props 类型
- 使用 forwardRef 支持 ref 转发
```

**设计意图**：当项目规模大到一个 CLAUDE.md 装不下时，拆分为模块化规则，并且支持按文件路径按需加载（不会一次性占满上下文窗口）。

#### L5：Local Memory（个人项目偏好层）

```
位置: ./CLAUDE.local.md
（自动加入 .gitignore）
```

个人在特定项目中的私有偏好，不进入版本控制。典型内容：

```markdown
# 我在本项目的偏好
- 我负责 payment 模块，优先关注 src/payment/
- 本地 DB 端口是 5433（不是默认 5432）
- 测试时跳过 e2e，只跑 unit: pnpm test:unit
```

**设计意图**：解决「项目规范是统一的，但我的本地环境/职责是特殊的」这个矛盾。L3 是团队共识，L5 是个人补充。

#### L6：Auto Memory（自动发现层）

```
位置: ~/.claude/projects/<project-hash>/memory/MEMORY.md
启动时加载: 前 200 行
```

这是唯一由 **Claude 自动维护** 的记忆层。Claude 在工作过程中发现的模式会被自动保存：

```markdown
# Auto Memory - my-project

## 项目架构
- 使用 monorepo 结构，packages/ 下有 3 个子包
- shared 包被其他两个包依赖

## 调试经验
- prisma migrate 失败时通常是 shadow database 端口冲突
- CI 中 vitest 需要 --pool=forks 参数

## 代码模式
- 所有 API handler 统一用 withAuth() 包装
- 错误处理使用自定义 AppError 类
```

**设计意图**：让 Claude 在反复工作中积累「项目直觉」。不需要人工维护，Claude 自己发现、自己记录。限制 200 行加载量是为了保护上下文窗口。

### 2.3 加载顺序与优先级

```
会话启动时的加载流程:

  ┌──────────────┐
  │ Session Start│
  └──────┬───────┘
         │
         ▼
  ┌──────────────────────────────────────────────┐
  │ 1. 加载 System Prompt (Anthropic 内置)        │
  └──────────────────────────┬───────────────────┘
                             │
  ┌──────────────────────────▼───────────────────┐
  │ 2. 加载 L1: Managed Policy                   │
  │    /etc/claude-code/CLAUDE.md                │
  └──────────────────────────┬───────────────────┘
                             │
  ┌──────────────────────────▼───────────────────┐
  │ 3. 加载 L2: User Memory                      │
  │    ~/.claude/CLAUDE.md                       │
  └──────────────────────────┬───────────────────┘
                             │
  ┌──────────────────────────▼───────────────────┐
  │ 4. 加载 L3: Project Memory                   │
  │    ./CLAUDE.md (当前目录及父目录)              │
  └──────────────────────────┬───────────────────┘
                             │
  ┌──────────────────────────▼───────────────────┐
  │ 5. 加载 L4: Project Rules (部分按需)          │
  │    ./.claude/rules/*.md                      │
  └──────────────────────────┬───────────────────┘
                             │
  ┌──────────────────────────▼───────────────────┐
  │ 6. 加载 L5: Local Memory                     │
  │    ./CLAUDE.local.md                         │
  └──────────────────────────┬───────────────────┘
                             │
  ┌──────────────────────────▼───────────────────┐
  │ 7. 加载 L6: Auto Memory (前 200 行)          │
  │    ~/.claude/projects/<hash>/memory/         │
  └──────────────────────────┬───────────────────┘
                             │
                             ▼
                    ┌────────────────┐
                    │ 会话就绪       │
                    │ 开始接收输入   │
                    └────────────────┘
```

**优先级规则**：更具体的层级覆盖更宽泛的层级。

```
优先级（高 → 低）:

  L5 Local Memory        最高（个人+项目特定）
  L4 Project Rules       ↑  （路径匹配的细粒度规则）
  L3 Project Memory      │  （项目团队共识）
  L2 User Memory         │  （个人全局偏好）
  L1 Managed Policy      最低（企业兜底策略）

  L6 Auto Memory 不参与覆盖，作为补充信息加载
```

**按需加载机制**：父目录的 CLAUDE.md 在启动时完整加载，子目录的 CLAUDE.md 和部分 Project Rules 在 Claude 导航到对应目录时才加载，避免启动时占用过多上下文。

## 3. 三层上下文压缩系统

当 6 层记忆加载完毕、会话开始运行后，上下文窗口会因为对话和工具调用不断增长。Claude Code 通过 3 层压缩系统管理运行时上下文。

### 3.1 上下文窗口布局

```
200K Token 上下文窗口 (2026年2月)
┌──────────────────────────────────────────────────┐
│                                                  │
│  System Prompt + 6 层记忆内容     (~5-15K)       │
│                                                  │
├──────────────────────────────────────────────────┤
│                                                  │
│  对话历史 + 工具调用结果          (动态增长)      │
│                                                  │
│  ← 可用空间随对话推进不断缩小 →                   │
│                                                  │
├──────────────────────────────────────────────────┤
│  Compaction Buffer (~33K)                        │
│  ├─ 压缩操作预留空间                             │
│  └─ 模型输出预留空间                             │
├──────────────────────────────────────────────────┤
│  Output Headroom                                 │
│  （模型生成 response 需要的空间）                  │
└──────────────────────────────────────────────────┘

可用对话空间 ≈ 200K - 记忆内容 - Buffer - Output
            ≈ 200K - 15K - 33K - 若干K
            ≈ ~150K 可用于对话和工具结果
```

### 3.2 Micro-compaction（微压缩）

**触发时机**：每次大输出的工具调用完成后立即触发。

**处理方式**：将大体积的工具输出（Read、Bash、Grep、Glob、WebSearch 等）转存到磁盘，上下文中只保留最近的「热尾部」（hot tail）结果。

```
工具调用结果处理:

  ┌───────────────────────────────────────┐
  │  Read file.ts → 2000 行输出           │
  └──────────────────┬────────────────────┘
                     │
            ┌────────▼────────┐
            │  是否为最近结果？ │
            └────────┬────────┘
               │          │
           最近的      较旧的
               │          │
               ▼          ▼
      ┌──────────┐  ┌──────────────┐
      │ 保留在   │  │ 转存磁盘     │
      │ 上下文中 │  │ 只保留引用   │
      └──────────┘  └──────────────┘
```

**设计意图**：大量工具输出是上下文膨胀的主要来源。通过及时转存，保证上下文空间留给真正需要推理的最新信息。

### 3.3 Auto-compaction（自动压缩）

**触发条件**：当已用上下文达到约 **83.5%**（即可用空间低于 Compaction Buffer）时自动触发。

```
触发阈值（2026年2月数据）:

  200K 总窗口
  ×  83.5% = ~167K 使用时触发
  
  Buffer = ~33K（占总窗口的 16.5%）
  
  历史演变:
  2025年中: Buffer ~45K, 触发于 ~77-78%
  2026年初: Buffer ~33K, 触发于 ~83.5%
  ↑ 随模型能力增强，压缩开销减少，可用空间增大
```

**压缩流程**：

```
Auto-compaction 流程:

  ┌──────────────────────┐
  │ 检测: 已用 ≥ 83.5%   │
  └──────────┬───────────┘
             │
             ▼
  ┌──────────────────────┐
  │ 1. 生成对话摘要       │
  │    (保留关键决策/文件) │
  └──────────┬───────────┘
             │
             ▼
  ┌──────────────────────┐
  │ 2. 清除旧消息         │
  │    (thinking tokens   │
  │     也被清除)         │
  └──────────┬───────────┘
             │
             ▼
  ┌──────────────────────┐
  │ 3. 重注水(Rehydrate) │
  │    ├─ 重新读取最近文件│
  │    ├─ 恢复 TODO 列表  │
  │    └─ 注入续接指令    │
  └──────────┬───────────┘
             │
             ▼
  ┌──────────────────────┐
  │ 4. 继续对话           │
  │    (几乎无感知中断)   │
  └──────────────────────┘
```

**重注水（Rehydration）** 是关键设计：压缩不只是丢弃信息，而是「摘要 + 恢复关键上下文」。压缩后 Claude 会重新读取最近操作的文件、恢复待办事项列表、注入继续执行的指令，确保任务不中断。

### 3.4 Manual-compaction（手动压缩）

**触发方式**：用户输入 `/compact` 命令，可附带焦点提示。

```bash
# 基本压缩
/compact

# 带焦点提示的压缩
/compact 保留关于 payment 模块重构的所有上下文
```

**适用场景**：
- 完成一个大任务后，准备开始新任务
- 想要重新聚焦，丢弃不相关的历史
- 上下文中积累了大量已解决问题的调试信息

### 3.5 三层压缩的协同关系

```
上下文使用量时间线:

  100% ┬─────────────────────────────────────────
       │
  83.5%├─ ─ ─ ─ ─ ─ ─ ─ ┬─ Auto-compaction 阈值
       │                 │
       │    ╱╲            │  ╱╲
       │   ╱  ╲  Micro    │ ╱  ╲  压缩后
       │  ╱    ╲ 持续     │╱    ╲ 使用量
       │ ╱      ╲ 削峰    │      ╲ 骤降
       │╱        ╲        │       ╲
       │          ╲       │        ╱╲
       │           ╲      │       ╱  ╲ 继续增长
       │            ╲─────│      ╱
       │                  │     ╱
    0% └──────────────────┴─────────────────────
       │← 正常工作阶段 →│←压缩→│←继续工作→│
       
  Micro-compaction: 持续削峰，延迟到达阈值
  Auto-compaction:  到达阈值时批量压缩
  Manual-compaction: 用户主动在任务边界清理
```

## 4. 与其他 Agent 的记忆系统对比

| 维度 | Claude Code | Codex | Gemini CLI | Kimi CLI | OpenCode |
|------|------------|-------|-----------|---------|----------|
| **持久化记忆层数** | 6 层 | 2 层 | 3 层 | 无专用层 | 2 层 |
| **记忆文件** | CLAUDE.md + rules + auto | instructions + AGENTS.md | GEMINI.md (3 层) | 无 | .opencode/rules |
| **企业策略层** | 有 (Managed Policy) | 无 | 无 | 无 | 无 |
| **自动记忆** | 有 (Auto Memory) | 无 | 有 (JIT 子目录) | 无 | 无 |
| **上下文窗口** | 200K | 取决于模型 | 取决于模型 | 取决于模型 | 取决于模型 |
| **压缩机制** | 3 层 (micro/auto/manual) | 无 | tryCompressChat | compact_context | SessionCompaction |
| **重注水** | 有 | 无 | 无 | 无 | 有（部分） |
| **回滚** | 无 | 无 | 无 | Checkpoint + D-Mail | 无 |

### 关键差异分析

**Claude Code 独有的设计**：

1. **企业策略层 (L1)**：唯一支持组织级 IT 管控的记忆层，其他 Agent 都没有
2. **Auto Memory (L6)**：Claude 自己发现并记录项目模式，跨会话积累「直觉」
3. **Micro-compaction**：其他 Agent 都是到达阈值才压缩，Claude Code 在每次大输出后就做微观削峰
4. **Rehydration**：压缩后主动恢复关键文件和任务列表，不只是丢弃旧信息

**Gemini CLI 的分层对比**：

```
Gemini CLI 3 层:           Claude Code 对应:
  L1 User-level GEMINI.md  ≈ L2 User Memory
  L2 Project GEMINI.md     ≈ L3 Project Memory
  L3 JIT 子目录加载         ≈ L4 Project Rules (按路径)
  
  缺失: 企业策略层、Local Memory、Auto Memory
```

**Kimi CLI 的替代方案**：

Kimi CLI 没有分层记忆文件，但它有 **Checkpoint + D-Mail**——通过「回到过去」解决记忆问题，而不是通过「分层预加载」。两种思路分别对应「恢复型」和「预防型」记忆管理。

## 5. 设计哲学

### 5.1 为什么是 6 层？

每一层对应一个真实的组织/项目场景：

```
企业 IT 要统一安全策略          → L1 Managed Policy
开发者有个人习惯                → L2 User Memory
项目团队有共识                  → L3 Project Memory
大项目需要模块化规则            → L4 Project Rules
每人本地环境不同                → L5 Local Memory
重复操作中的经验值得记住        → L6 Auto Memory
```

### 5.2 为什么用 Markdown 文件而不是数据库？

Claude Code 的设计哲学是「简单优先」：

| 选项 | 优势 | 劣势 | Claude Code 的选择 |
|------|------|------|-------------------|
| Markdown 文件 | 人可读、可编辑、可 Git | 查询能力弱 | 选择 |
| SQLite | 结构化查询 | 不可 Git、人不可直接编辑 | 不选 |
| Embedding 向量库 | 语义检索 | 复杂、不可解释 | 不选 |

Markdown 文件的核心优势：**人和 AI 都能读写**。团队成员可以直接编辑 CLAUDE.md，不需要任何特殊工具。这与 Claude Code 整体的「原始工具组合」哲学一致——用最简单的方式解决问题。

### 5.3 上下文即稀缺资源

Claude Code 的架构将 200K token 上下文窗口视为最宝贵的资源。6 层记忆 + 3 层压缩的设计都围绕一个核心问题：**如何在有限的窗口中维持最大的有效信息密度**。

```
信息来源          管理策略

  6 层记忆内容      按需加载（子目录/rules 延迟加载）
  工具输出          微压缩（大输出立即转存磁盘）
  对话历史          自动压缩（83.5% 时触发）
  thinking tokens   自动清除（前几轮的 thinking 不保留）
  最终 fallback     用户手动 /compact 清理
```

## 6. 实践建议

### 6.1 记忆层的最佳配置

```
推荐的记忆分层策略:

  L1 (Managed Policy)  → 企业安全红线、合规要求
  L2 (User Memory)     → 语言偏好、格式习惯（保持简短 <50 行）
  L3 (Project Memory)  → 技术栈、核心命令、架构概述（<200 行）
  L4 (Project Rules)   → 按模块拆分详细规范（每个 <100 行）
  L5 (Local Memory)    → 本地端口、个人职责（<30 行）
  L6 (Auto Memory)     → 不需手动管理，但可定期清理过时内容
```

### 6.2 保护上下文的技巧

1. **CLAUDE.md 要精简**：每多写一行，可用对话空间就少一点
2. **用 Project Rules 替代超长 CLAUDE.md**：按路径匹配实现按需加载
3. **大文件让 Claude 自己读**：不要粘贴到对话中，让工具调用去读
4. **任务边界用 `/compact`**：完成一个任务后手动压缩再开始下一个
5. **利用 Auto Memory**：重复解释的内容，让 Claude 自动记住
