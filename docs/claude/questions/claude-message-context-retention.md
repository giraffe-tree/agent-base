# Claude Code 消息上下文保留机制

**结论**：Claude Code 在工具执行后将结果传回模型时，**会完整保留**之前消息中的 `reasoning_content`，确保推理链条的连贯性。

---

## 消息结构

Claude Code 使用多模态消息格式，每条助手消息可包含多种 content block：

```
┌─────────────────────────────────────────┐
│  Assistant Message                      │
├─────────────────────────────────────────┤
│  ┌─────────────────────────────────┐    │
│  │  thinking (reasoning_content)   │    │
│  │  "让我分析这个问题..."           │    │
│  └─────────────────────────────────┘    │
│  ┌─────────────────────────────────┐    │
│  │  tool_use                       │    │
│  │  调用: Read<file_path="...">    │    │
│  └─────────────────────────────────┘    │
└─────────────────────────────────────────┘
```

---

## 工具执行后的消息流

### 完整流程

```
┌──────────────┐     ┌──────────────┐     ┌──────────────┐
│   用户输入    │────▶│  模型推理+   │────▶│   系统执行   │
│              │     │  工具调用    │     │    工具      │
└──────────────┘     └──────────────┘     └──────┬───────┘
                                                  │
                                                  ▼
┌──────────────┐     ┌──────────────┐     ┌──────────────┐
│   模型继续    │◀────│  完整上下文  │◀────│  工具结果    │
│   推理输出    │     │  (含reasoning)│    │  追加到历史  │
└──────────────┘     └──────────────┘     └──────────────┘
```

### 消息历史示例

```json
[
  {
    "role": "assistant",
    "content": [
      {
        "type": "thinking",
        "thinking": "让我分析这个问题。首先需要读取配置文件..."
      },
      {
        "type": "tool_use",
        "name": "Read",
        "input": {"file_path": "/path/to/config.json"}
      }
    ]
  },
  {
    "role": "user",
    "content": [
      {
        "type": "tool_result",
        "content": "{\"key\": \"value\"}"
      }
    ]
  }
]
```

---

## 关键设计要点

| 特性 | 说明 |
|------|------|
| **推理连续性** | reasoning_content 作为消息历史的一部分参与上下文构建 |
| **工具结果追加** | 新工具结果以 `tool_result` block 形式追加，不覆盖历史 |
| **上下文完整性** | 下一次模型调用能访问完整的推理链条和工具调用历史 |

---

## 与其他 Agent 的对比

| 项目 | 推理保留机制 | 特点 |
|------|-------------|------|
| **Claude Code** | 原生保留 reasoning_content | 推理过程对用户可见，支持多轮工具调用 |
| OpenCode | 内部状态维护 | 依赖 TypeScript runtime 的状态管理 |
| Gemini CLI | 通过 scheduler state | 状态机驱动，推理中间状态保存在 scheduler |
| Kimi CLI | 显式 checkpoint | 通过 D-Mail 机制持久化关键状态 |

---

## 实际意义

1. **多轮工具调用**：模型可以在之前的推理基础上决定下一步操作
2. **错误恢复**：即使工具执行失败，模型仍能看到之前的推理逻辑
3. **连贯体验**：用户可以看到完整的思考过程，增强可解释性
